# 算法完整流程与单步计算全过程（按当前代码梳理）

这份文档只按当前仓库代码梳理，不按论文原始写法复述。

先给出三个最核心的事实：

1. **网络只预测步长 `ds`**
2. **推理执行器是 `ContourTracker` 的 tangent tracker**
3. **真正的性能收益来自减少昂贵刷新频率和总 rollout 步数，而不是消灭 SVD**

---

### 1.1 数据生成链

对应文件：

- `scripts/generate_large_dataset.py`
- `src/train/expert_solver.py`
- `src/train/dagger_augmentation.py`

作用：生成 `(features, ds_expert)` 监督样本。

### 1.2 训练链

对应文件：

- `scripts/train_from_dataset.py`
- `src/nn/controller.py`
- `src/nn/loss.py`

作用：训练一个从 8 维特征到步长 `ds` 的回归器。

### 1.3 推理链

对应文件：

- `scripts/run_tracking.py`
- `src/core/contour_tracker.py`
- `src/nn/inference_controller.py`

作用：网络给步长，tracker 真正执行 contour rollout。

## 2. 当前代码里的状态变量

不管是专家链还是推理链，核心状态都是：

- `z`：复平面上的当前位置
- `u`：最小奇异值对应的左奇异向量
- `v`：最小奇异值对应的右奇异向量

为了做步长控制，推理还保留两个历史量：

- `prev_ds`
- `prev_applied_projection`

所以每一步控制器实际面对的局部状态可以写成：

```text
(z, u, v, prev_ds, prev_applied_projection)
```

但网络并不直接吃这 5 个对象，而是吃从它们提出来的 8 维归一化特征。

---

## 3. 网络真正输入的 8 维是什么

对应代码：

- `src/nn/features.py`
- `src/core/contour_tracker.py`

### 3.1 语义层面的 8 个量

按顺序是：

1. `|sigma_approx - epsilon|`
2. `||Mv - epsilon u||`
3. `|u^*v|`
4. `|epsilon|`
5. `matrix_scale = ||A||_F / sqrt(n)`
6. `|z| / matrix_scale`
7. `prev_ds`
8. `prev_applied_projection`

其中：

- `M = zI - A`
- `sigma_approx = |u^*Mv|`

### 3.2 网络真正看到的是归一化值

前 7 个连续量都会经过 `_log_normalize(...)`，压到 `[-1, 1]` 附近：

\[
\mathrm{lognorm}(x;a,b)=\mathrm{clip}\left(2\cdot\frac{\log_{10}(\max(x,10^a))-a}{b-a}-1,-1,1\right)
\]

当前默认参数是：

- 第 1、2 维：`lognorm(x; -12, 0)`
- 第 3 维：`lognorm(x; -6, 0)`
- 第 4 维：`lognorm(x; -12, 2)`
- 第 5、6 维：`lognorm(x; -6, 3)`
- 第 7 维：`lognorm(x; -8, -1)`
- 第 8 维：直接是 `0/1`

所以严格来说，MLP 输入是：

```text
normalized_features ∈ R^8
```

不是原始物理量。

## 4. 数据生成全流程

当前数据集不是凭空标注的，而是通过“专家 + teacher-forced tracker + DAgger”生成。

### 4.1 随机生成任务

`scripts/generate_large_dataset.py` 会：

1. 随机选矩阵类型
2. 生成矩阵 `A`
3. 随机采样复平面点 `z_random`
4. 计算
   \[
   \epsilon = \sigma_{\min}(z_{random}I-A)
   \]
5. 把经过该点的这条 `epsilon` contour 当成当前任务

所以每个任务本质上是：

- 一个矩阵 `A`
- 一个层级 `epsilon`
- 一个起点 `z0`

### 4.2 专家给出步长标签

`src/train/expert_solver.py` 负责高精度 teacher。

专家的核心逻辑是：

1. 用 full triplet RK4 推进 `(z,u,v)`
2. 在候选点做 exact SVD
3. 如果离 contour 仍有偏差，先试 local projection
4. local 不够时，再做 radial projection
5. 得到被接受的一小步
6. 记下这一步的长度 `ds_expert`

所以专家输出的不是方向，而是：

- `ds_expert`

方向依然由白盒几何给出。

### 4.3 Teacher-forced tracker 对齐推理分布

数据脚本不会直接把专家内部状态当最终训练样本，而是还会做一件非常关键的事：

- 用专家给出的 `ds_expert`
- 调用和推理时同构的 `ContourTracker._advance_step(...)`
- 让 tracker 真正执行这一步

这样记录出来的数据更接近真实推理分布，而不是只停留在理想化专家分布。

### 4.4 DAgger 增强

`src/train/dagger_augmentation.py` 会：

1. 对轨迹上的 `(z,u,v)` 加小扰动
2. 尽量投影回 contour 附近
3. 重新问专家：在这个偏离状态下应该走多大步
4. 生成额外的恢复样本

### 4.5 数据落盘

每个样本最终会写成：

- `features`
- `ds_expert`
- `epsilon`
- `matrix_size`
- `matrix_type`
- `matrix_id`
- `trajectory_id`
- `source`

其中：

- `features` 已经是归一化后的 8 维向量
- `source` 是 `expert` 或 `dagger`

## 5. 训练全流程

训练链路很干净：

1. `src/data/dataset.py` 读入 `features` 和 `ds_expert`
2. `src/nn/controller.py` 里的 MLP 输出 `ds_pred`
3. `src/nn/loss.py` 计算
   \[
   (\log ds_{pred}-\log ds_{expert})^2
   \]
4. 用 AdamW 训练
5. 保存 `best_model.pt`

### 5.1 控制器结构

默认控制器是一个 MLP：

- 输入维度：`8`
- hidden dims：默认 `[128, 128, 64]`
- norm：`layernorm`
- activation：`silu`
- head hidden dim：默认 `64`

### 5.2 输出如何映射成步长

`NNController.forward(...)` 的输出不是裸实数，而是一个始终为正的步长。

在默认配置下，`step_size_max` 是存在的，因此代码使用：

\[
\exp(\log s_{min} + \sigma(\text{raw}) (\log s_{max} - \log s_{min}))
\]

所以网络输出会被限制在：

```text
[step_size_min, step_size_max]
```

默认是：

- `step_size_min = 1e-4`
- `step_size_max = 1e-1`

## 6. 推理全流程

推理对应：

- `scripts/run_tracking.py`
- `src/core/contour_tracker.py`
- `src/nn/inference_controller.py`

### 6.1 初始化

给定起点 `z0` 后，`run_tracking.py` 会：

1. 读入矩阵 `A`
2. 确定 `epsilon`
3. 如果用户直接给了 `z0`，先用 `project_to_contour(...)` 把它投影到 contour 上
4. 如果用户没给 `z0`，就用 `auto_select_contour_start(...)` 自动找起点
5. 在起点做一次精确 SVD，得到 `(u0, v0)`
6. 初始化 tracker 状态

### 6.2 主循环

`ContourTracker.track(...)` 每一步都做：

1. 从当前状态提取 8 维特征
2. 控制器预测 `ds`
3. `AdaptiveInferenceController` 对 `ds` 做稳定化包装
4. 调用 `_advance_step(...)` 执行这一步
5. 更新轨迹、几何统计量和日志
6. 调用 `check_closure(...)` 判断是否闭合

如果没有提供 checkpoint，控制器就是 `None`，这时 `ContourTracker` 会直接使用固定步长 `fixed_step_size`。

## 7. 单步计算全过程

这一节只讲最关键的部分：

- `ContourTracker._advance_step(...)`

设当前状态为：

```text
(z_t, u_t, v_t)
```

控制器给出的步长为：

```text
ds_t
```

下面按代码实际顺序展开。

### 7.1 先确定真正执行的步长

代码第一步会做：

```text
ds_step = max(ds_t, min_step_size)
```

也就是说，tracker 至少会保证这一步不小于系统允许的最小步长。

### 7.2 用白盒切向公式做 predictor

切向方向由：

\[
\frac{dz}{ds}= i\frac{v^*u}{|v^*u|}
\]

代码对应：

- `ContourTracker._compute_tangent_direction(u, v)`

于是：

```text
gamma = v_t^* u_t
dz_ds = i * gamma / |gamma|
z_candidate = z_t + ds_step * dz_ds
```

这一快路径里，triplet 会先直接复用：

```text
u_candidate = u_t
v_candidate = v_t
```

这一步本身很便宜，真正贵的是后面要不要做精确刷新和投影。

### 7.3 判断是否允许 approximate triplet 快路径

代码里有一个条件：

```text
can_use_approx_triplet =
    exact_triplet_refresh_interval > 0
    and steps_since_exact_triplet_refresh < exact_triplet_refresh_interval
    and approx_triplet_sigma_tol is not None
    and approx_triplet_residual_tol is not None
```

默认参数下：

- `exact_triplet_refresh_interval = 8`
- `approx_triplet_sigma_tol = 5e-3`
- `approx_triplet_residual_tol = 5e-2`

也就是说，在最近若干步没有强制 exact refresh 的窗口里，tracker 会优先尝试便宜的 approximate triplet 评估。

### 7.4 approximate triplet 指标怎么计算

如果允许走快路径，代码会算：

```text
Mv = (z_candidate I - A) v_candidate
sigma_approx = |u_candidate^* Mv|
residual_approx = ||Mv - epsilon u_candidate||
sigma_error_approx = |sigma_approx - epsilon|
```

然后再估一个局部投影距离：

```text
approx_projection_distance = sigma_error_approx / |v_candidate^* u_candidate|
```

以及一个 residual 门限：

```text
residual_limit = max(approx_triplet_residual_tol, 1.25 * max(ds_step, min_step_size))
```

### 7.5 分支 A：approximate 直接接受

如果同时满足：

```text
sigma_error_approx <= approx_triplet_sigma_tol
residual_approx <= residual_limit
```

那么当前这一步直接接受：

```text
(z_{t+1}, u_{t+1}, v_{t+1})
= (z_candidate, u_candidate, v_candidate)
```

诊断里会标成：

- `projection_mode = "none"`
- `triplet_refresh_mode = "approx_skip"`
- `exact_triplet_refresh = False`

这是最便宜的一种 accepted step。

### 7.6 分支 B：approximate deferred accept

代码还允许一个更宽松的窗口：

```text
relaxed_projection_tol = projection_tol * projection_defer_factor
```

默认：

- `projection_tol = 1e-4`
- `projection_defer_factor = 4.0`
- 所以 relaxed tol 默认是 `4e-4`

如果同时满足：

```text
max_deferred_projection_steps > 0
deferred_projection_streak < max_deferred_projection_steps
sigma_error_approx <= relaxed_projection_tol
approx_projection_distance <= projection_defer_distance_ratio * max(ds_step, min_step_size)
residual_approx <= 2 * residual_limit
```

那么当前代码会先接受这一步：

```text
(z_{t+1}, u_{t+1}, v_{t+1})
= (z_candidate, u_candidate, v_candidate)
```

但会把它标成：

- `projection_mode = "deferred_accept"`
- `triplet_refresh_mode = "approx_deferred_accept"`

意思是：

- 这一步没有精确纠偏
- 但误差还在可控区间内
- 可以先把精确刷新往后拖

### 7.7 分支 C：approximate local projection

如果 approximate 误差已经不能直接接受，但还在 local projection 能处理的范围内，代码会尝试：

- `project_to_contour_by_local_normal(...)`

触发窗口是：

```text
approx_projection_window = max(6 * approx_triplet_sigma_tol,
                               3 * relaxed_projection_tol)
```

再要求：

```text
sigma_error_approx <= approx_projection_window
residual_approx <= 3 * residual_limit
```

local projection 内部会：

1. 复用当前 `sigma/u/v`
2. 沿估计法向做最多 `max_newton_iters=2` 次局部 Newton 型修正
3. 只要误差明显下降并进入 `projection_tol`，就接受

如果成功，返回：

- `projection_mode = "local_normal_newton"` 或 `"local_exact"`
- `triplet_refresh_mode = "approx_local_projection"`
- `applied_projection = True`
- `exact_triplet_refresh = True`

### 7.8 分支 D：做 exact SVD refresh

如果 approximate 路径没有直接解决问题，代码就会在 `z_candidate` 上做一次精确 SVD：

```text
sigma_candidate, u_exact, v_exact = svd_solver(A, z_candidate)
```

然后标准化 `u_exact, v_exact`，并计算：

```text
sigma_error = |sigma_candidate - epsilon|
```

#### D1. exact 直接接受

如果：

```text
sigma_error <= projection_tol
```

那么直接接受：

```text
(z_{t+1}, u_{t+1}, v_{t+1}) = (z_candidate, u_exact, v_exact)
```

并记录：

- `projection_mode = "none"`
- `triplet_refresh_mode = "exact_svd"`

#### D2. exact deferred accept

如果偏差虽然没有压到 `projection_tol`，但仍在 deferred window 内：

```text
sigma_error <= relaxed_projection_tol
estimated_projection_distance <= projection_defer_distance_ratio * max(ds_step, min_step_size)
```

那么代码也会先接受 exact triplet：

```text
(z_{t+1}, u_{t+1}, v_{t+1}) = (z_candidate, u_exact, v_exact)
```

记录：

- `projection_mode = "deferred_accept"`
- `triplet_refresh_mode = "exact_svd"`

这和 approximate deferred accept 的区别在于：

- 这里已经花了一次 exact SVD
- 但没有继续追加 projection

#### D3. exact local projection

如果 exact 误差不够小，代码会再次尝试 local projection：

- `project_to_contour_by_local_normal(...)`

此时使用的是更可靠的 `u_exact, v_exact`。

如果成功，就接受投影结果。

#### D4. radial projection fallback

如果 local projection 还不够，最后再调：

- `project_to_contour(...)`

这一步会把点全局拉回 contour。

成功后还会对投影点重新刷新 triplet：

```text
_, u_projected, v_projected = refresh_triplet(z_projected)
```

诊断标记为：

- `projection_mode = "radial_fallback"`
- `triplet_refresh_mode = "radial_projection"`

#### D5. exact fallback

如果连 radial projection 都没给出可接受结果，当前代码会直接接受：

```text
(z_{t+1}, u_{t+1}, v_{t+1}) = (z_candidate, u_exact, v_exact)
```

并记录：

- `projection_mode = "exact_fallback"`
- `triplet_refresh_mode = "exact_svd_fallback"`

也就是说，这个版本的最终兜底是：

- 保留这一步 exact triplet
- 把后续稳定性交给后面的控制器和下一步 rollout

### 7.9 单步函数的返回值

`_advance_step(...)` 返回：

```text
z_next, u_next, v_next, accepted_ds, step_diagnostics
```

其中 `step_diagnostics` 里会带很多重要字段，例如：

- `sigma`
- `sigma_error`
- `raw_sigma`
- `raw_sigma_error`
- `projection_distance`
- `projection_mode`
- `triplet_refresh_mode`
- `triplet_residual`
- `exact_triplet_refresh`

## 8. 一小步返回后还会发生什么

`ContourTracker.track(...)` 在接收这一小步之后，还会继续做几件事。

### 8.1 更新 deferred / refresh 计数

如果这一步：

- 做了 projection，则 `deferred_projection_streak = 0`
- 是 `deferred_accept`，则 streak 加 1
- 做了 exact refresh，则 `steps_since_exact_triplet_refresh = 0`
- 否则 refresh 计数加 1

### 8.2 更新路径几何量

会累计：

- `step_distance`
- `path_length`
- `max_distance_from_start`
- `winding_angle`

这些量不进入网络，但会用于闭合判断和日志。

### 8.3 计算 `tangent_turn`

代码会比较当前步和上一小步的切向角度变化，得到：

- `tangent_turn`

这个量不进 MLP，但会交给 `AdaptiveInferenceController.observe_step(...)`，用于判断局部曲率是否变大。

### 8.4 更新 `TrackerState`

新状态会写成：

```text
state = (z_next, u_next, v_next, prev_ds=accepted_ds,
         prev_applied_projection=applied_projection)
```

下一步提特征时，就会把这两个历史量一起带进去。

### 8.5 写日志 / callback

step callback 会记录：

- `raw_ds`
- `ds`
- `sigma_error`
- `raw_sigma_error`
- `applied_projection`
- `projection_mode`
- `triplet_refresh_mode`
- `tangent_turn`
- `features`
- `controller_info`

这些字段会写入 `tracking_steps.jsonl` 或 benchmark 日志。

## 9. `AdaptiveInferenceController` 到底做什么

`src/nn/inference_controller.py` 不是另一个求解器，而是网络输出外面的一层轻量包装。

### 9.1 预测阶段

`predict_with_info(...)` 做三件事：

1. 调基础 MLP 得到 `ds`
2. 用 `min_step_size` 和动态 ceiling 做 clamp
3. 如果最近连续很多步都很稳定，就乘上一个增长系数

默认参数下：

- `stable_growth_factor = 1.25`
- `stable_growth_interval = 4`

所以如果连续稳定，未来步长 ceiling 会逐渐放松。

### 9.2 观测阶段

`observe_step(...)` 会读入：

- `applied_projection`
- `raw_sigma_error`
- `projection_distance`
- `tangent_turn`
- `ds`

然后做两类惩罚：

#### 投影相关惩罚

如果：

- 投影过于频繁
- 或投影距离相对步长过大
- 或原始 sigma 偏差过大

就收紧 dynamic ceiling。

#### 曲率相关惩罚

如果 `tangent_turn` 连续偏大，也会收紧 dynamic ceiling。

所以网络的原始输出并不会被生硬直接使用，而是经过一层推理期稳定化。

## 10. 闭合判定怎么做

`ContourTracker.check_closure(...)` 不是只看“当前点是不是回到起点附近”。

它会同时要求：

1. 已经走了足够多步
2. `path_length` 足够长
3. `max_distance_from_start` 足够大
4. `winding_angle` 足够大
5. 当前点或当前线段离起点足够近

其中实际闭合阈值还会动态放宽成：

```text
effective_closure_tol = max(closure_tol, 0.5 * last_step_size)
```

这样可以避免数值上过早把局部回头误判成闭合。

## 11. 为什么“还是会做 SVD”却仍然可能有效

这是整个项目最关键的问题。

问题可以表述成：

> 传统算法的 bottleneck 是 SVD 很慢。
> 当前方法并没有把 SVD 消掉，为什么仍然可能更快？

答案是：**收益来自改变整条 rollout 的计算结构。**

### 11.1 方向本来就便宜

当前切向 predictor 只用：

- `v^*u`
- 一个复数归一化
- 一次显式欧拉式位置更新

这部分很便宜，本来就不是主要瓶颈。

### 11.2 关键在于减少 expensive refresh 的频率

当前 tracker 有多层过滤：

1. approximate triplet 快路径
2. deferred accept
3. local projection
4. exact SVD refresh
5. radial projection

并不是每一步都直接跳到第 4、5 层。

如果网络把步长控制得更稳，就会有更多步停在前 1~3 层，昂贵刷新出现的次数就下降。

### 11.3 关键还在于减少总步数

如果步长策略足够好：

- 平滑区域敢走大步
- 风险区域及时保守

那么同一条 contour 的 accepted steps 会减少。

即使某些步仍然要做 SVD，只要**总步数更少、贵步骤比例更低**，整体 wall-clock 仍然可能下降。

### 11.4 teacher 和 inference 的角色不同

需要分清两件事：

- **专家链**：允许更贵，因为它是离线标注器
- **推理链**：要尽量把贵步骤推迟到必要时刻

当前方法真正优化的是推理链，而不是离线 teacher 的单步代价。
