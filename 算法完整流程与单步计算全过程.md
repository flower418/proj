# 算法完整流程与单步计算全过程（当前代码版）

这份文档**只按当前代码分析**，不按论文原始写法复述。

当前版本的三个结论先写在最前面：

1. **网络只预测步长 `ds`**
2. **推理主链已经没有 restart 分支，也没有多轮缩步重试分支**
3. **真正跑推理的是单一 tangent tracker，不是多 ODE 方法切换**

## 2. 当前状态变量到底是什么

不管是专家链还是推理链，核心状态都是：

- `z`：复平面上的当前位置
- `u`：最小奇异值对应的左奇异向量
- `v`：最小奇异值对应的右奇异向量

推理控制还会额外带两个历史量：

- `prev_ds`
- `prev_applied_projection`

所以当前控制器看到的“局部状态”实际上是：

```text
(z, u, v, prev_ds, prev_applied_projection)
```

但网络并不直接吃这些原始量，而是吃从它们提取出来的 8 维特征。

---

## 3. 网络真正输入的 8 维是什么

对应代码：

- `src/nn/features.py`
- `src/core/contour_tracker.py`

按顺序是：

1. `|sigma_approx - epsilon|`
2. `||Mv - epsilon u||`
3. `|u^*v|`
4. `|epsilon|`
5. `matrix_scale`
6. `|z| / matrix_scale`
7. `prev_ds`
8. `prev_applied_projection`

其中：

- `M = zI - A`
- `sigma_approx = |u^*Mv|`
- `matrix_scale = ||A||_F / sqrt(n)`

这 8 维里：

- 前 3 维看当前 triplet 靠不靠谱
- 中间 3 维看任务尺度和当前位置
- 最后 2 维看上一步控制反馈

## 4. 数据生成全流程

当前数据集不是凭空标注出来的，而是这样生成：

### 第一步：随机生成任务

`scripts/generate_large_dataset.py` 会：

1. 随机选矩阵类型
2. 生成矩阵 `A`
3. 随机采样复平面点 `z_random`
4. 计算
   \[
   \epsilon = \sigma_{\min}(z_{random}I-A)
   \]
5. 把经过 `z_random` 的这条 `epsilon` 等高线当成任务

所以每个任务天然是：

- 一个矩阵 `A`
- 一个等高线层级 `epsilon`
- 一个起点 `z0`

### 第二步：专家给出步长标签

`src/train/expert_solver.py` 负责高精度 teacher。

专家并不是输出方向，而是输出：

- `ds_expert`

专家内部做的是：

1. 用 full triplet RK4 推进 `(z,u,v)`
2. 到候选点后做 exact SVD
3. 如果偏离 contour 太多，就先尝试局部法向投影
4. 还不够稳，再做全局径向投影
5. 得到一个被接受的小步及其终点
6. 把这个被接受的小步长度记成 `ds_expert`

### 第三步：teacher-forced tracker 复现推理分布

数据脚本不会直接拿专家状态当最终样本，而是还会走一层 teacher-forced tracker：

- 用专家给出的 `ds_expert`
- 交给和推理同构的 `ContourTracker`
- 让 tracker 真正执行这一步

这样记录下来的状态分布更接近真实推理分布，而不只是理想专家分布。

### 第四步：提取训练样本

每个样本最后写成：

- `features`
- `ds_expert`
- `epsilon`
- `matrix_size`
- `matrix_type`
- `matrix_id`
- `trajectory_id`
- `source`

其中 `source` 是：

- `expert`
- `dagger`

### 第五步：DAgger 增强

`src/train/dagger_augmentation.py` 会对轨迹点加小扰动，再重新问专家：

- 如果推理状态稍微偏掉，应该给多大的恢复步长

这会让网络更适应真实 rollout 中的不完美状态。

## 5. 训练全流程

训练链比现在已经非常干净：

1. 从数据集里读入 `features`
2. 网络输出一个标量 `ds_pred`
3. 与 `ds_expert` 做回归
4. 当前主要是 log-space MSE 监督

也就是说当前网络学的是：

```text
8维局部状态  ->  下一步该走多远
```

不是分类，不是 restart，不是多头策略。

---

## 6. 推理全流程

推理对应：

- `scripts/run_tracking.py`
- `src/core/contour_tracker.py`
- `src/nn/inference_controller.py`

它的完整流程如下。

### 6.1 初始化

给定起点 `z0` 后：

1. 先检查 `z0` 是否已经在 contour 上
2. 如果不在，就先投影到 `sigma_min(zI-A)=epsilon`
3. 在起点做一次精确 SVD，拿到 `(u0,v0)`
4. 初始化状态：
   - `z = z0`
   - `u = u0`
   - `v = v0`
   - `prev_ds = 0`
   - `prev_applied_projection = False`

### 6.2 每一步主循环

每一步都做以下事情：

1. 提 8 维特征
2. 控制器预测 `ds`
3. `AdaptiveInferenceController` 对 `ds` 做稳定化包装
4. tracker 执行这一小步
5. 更新状态与日志
6. 检查是否闭合

## 7. 单步计算全过程

下面是最关键的一段：**当前代码里一小步到底怎么被算出来。**

对应入口：

- `ContourTracker._advance_step(...)`

设当前状态为：

```text
(z_t, u_t, v_t)
```

控制器给出的步长为：

```text
ds_t
```

下面逐步展开。

### 7.1 先把步长夹到合法范围

代码里会先做：

```text
ds_step = max(ds_t, min_step_size)
```

也就是先保证：

- 步长不会小于系统允许的最小值

当前推理主链不会在这一层再做缩步重试；这一小步就是按这个 `ds_step` 执行。

### 7.2 用白盒切向公式做 predictor

当前切向方向是：

\[
\frac{dz}{ds}= i\frac{v^*u}{|v^*u|}
\]

代码实现对应：

- `ContourTracker._compute_tangent_direction(u, v)`

因此先计算：

```text
gamma = v^* u
dz_ds = i * gamma / |gamma|
```

然后得到预测点：

```text
z_candidate = z_t + ds_step * dz_ds
```

同时当前快路径会直接复用旧 triplet：

```text
u_candidate = u_t
v_candidate = v_t
```

注意这里就能看出一个非常关键的事实：

- **切向推进本身非常便宜**
- 真正贵的是后面“要不要做精确刷新 / 投影”的判断与修正

### 7.3 先尝试近似 triplet 判定

如果还没有到必须精确刷新的时刻，tracker 会先做一次便宜判定：

- `sigma_approx = |u^* (z_candidate I - A) v|`
- `residual = ||(z_candidate I - A)v - epsilon u||`

对应代码：

- `ContourTracker._approximate_triplet_metrics(...)`

然后构造几个量：

```text
sigma_error_approx = |sigma_approx - epsilon|
projection_distance_est ≈ sigma_error_approx / |v^*u|
```

这里的直觉是：

- `sigma_error_approx` 看当前点离 contour 偏了多少
- `residual` 看复用的 `(u,v)` 还可信不可信
- `projection_distance_est` 粗略估计如果纠偏，大概要挪多远

### 7.4 情况 A：近似上已经足够好，直接接受

如果同时满足：

- `sigma_error_approx` 很小
- `residual` 也很小

那这一小步直接接受：

```text
(z_{t+1}, u_{t+1}, v_{t+1})
= (z_candidate, u_candidate, v_candidate)
```

此时：

- 没有做精确 SVD
- 没有做投影
- 这一步最便宜

代码里会把它标成：

- `triplet_refresh_mode = "approx_skip"`
- `applied_projection = False`

### 7.5 情况 B：虽然不够精确，但允许延迟纠偏

如果近似误差不算特别小，但还在可接受窗口内，而且估计纠偏距离也不大，那么当前代码会允许：

- 暂时先接受
- 下一两步再看是否需要真正纠偏

这就是所谓 deferred projection。

此时仍然返回：

```text
(z_candidate, u_candidate, v_candidate)
```

只是在诊断里标记：

- `projection_mode = "deferred_accept"`

它的作用是：

- 避免每一步都急着上 SVD / projection
- 在平滑区域把快路径尽量拉长

### 7.6 情况 C：近似不够稳，先试局部投影

如果近似误差已经不能放心接受，但还处在局部法向修正能处理的范围内，tracker 会调用：

- `project_to_contour_by_local_normal(...)`

这个函数现在的逻辑很简单：

1. 用当前 `(u,v)` 估计 contour 的局部法向
2. 沿法向做少量 Newton 型修正
3. 如果误差明显下降并压到容差内，就接受投影后的点
4. 否则返回 `None`

也就是说当前局部投影是：

- **少量直接局部修正**
- **成功就用**
- **不成功就立刻升级到更贵方案**

如果局部投影成功，那么：

```text
(z_{t+1}, u_{t+1}, v_{t+1}) = local_projected_state
```

并且：

- `applied_projection = True`

### 7.7 情况 D：做一次精确 SVD 再判断

如果上面近似路径不能直接决定，tracker 就会在 `z_candidate` 做一次精确 SVD：

```text
sigma_candidate, u_exact, v_exact = svd_solver(A, z_candidate)
```

得到：

- 精确最小奇异值
- 精确左右奇异向量

然后重新判断：

#### D1. 精确上已经在 contour 附近

如果：

```text
|sigma_candidate - epsilon| <= projection_tol
```

那么直接接受：

```text
(z_{t+1}, u_{t+1}, v_{t+1}) = (z_candidate, u_exact, v_exact)
```

#### D2. 精确上允许延迟纠偏

如果虽然不够精确，但偏差还比较温和，也允许先接受 exact triplet 状态，暂时不投影。

这一步的意义是：

- 你已经花了一次 SVD
- 但还不急着立刻再做 projection

#### D3. 精确上做局部投影

如果 exact triplet 已经拿到了，那么局部投影的几何信息也更可靠。

于是再尝试一次局部法向投影：

- 成功则接受投影后的点
- 不成功则继续升级

### 7.8 情况 E：最后做全局径向投影

如果局部法向修正还是不够，最后再调用：

- `project_to_contour(...)`

这一步是更全局、更稳但也更重的修正。

如果它成功把点拉回 contour：

```text
(z_{t+1}, u_{t+1}, v_{t+1}) = radial_projected_state
```

并重新刷新 triplet。

### 7.9 情况 F：都不理想时的最终兜底

如果：

- 近似接受不放心
- 局部投影失败
- 全局投影也没给出可接受结果

当前代码不会再做多轮缩步重试。

它会直接返回：

- 当前这次 exact SVD 对应的 `z_candidate`
- 对应的精确 `u_exact, v_exact`

也就是说，当前主链的兜底是：

- **接受这次 exact candidate**
- **把后续稳定性交给后面的控制器与后续步处理**

这就是当前版本和旧版最重要的结构差异之一。

## 8. 一小步返回后，tracker 还会做什么

一旦 `_advance_step(...)` 返回，当前小步就被接受了。

这时主循环还会继续做下面这些更新。

### 8.1 更新历史状态

更新：

- `z`
- `u`
- `v`
- `prev_ds = accepted_ds`
- `prev_applied_projection = applied_projection`

### 8.2 更新路径几何量

会继续累计：

- `step_distance`
- `path_length`
- `max_distance_from_start`
- `winding_angle`
- `tangent_turn`

这些量不直接给网络输入，但会用于：

- 闭合判断
- 运行诊断
- 自适应控制器反馈

### 8.3 把这一步结果反馈给 `AdaptiveInferenceController`

`AdaptiveInferenceController.observe_step(...)` 主要看：

- `applied_projection`
- `raw_sigma_error`
- `projection_distance`
- `tangent_turn`

它会做三类事情：

1. 如果最近一直很稳，就允许未来步长 ceiling 慢慢放大
2. 如果投影过于频繁或投影距离过大，就收紧 ceiling
3. 如果切向变化过快，说明局部曲率偏大，也会收紧 ceiling

所以网络输出的原始 `ds` 并不会被生硬地直接拿去跑，外面还有一层轻量但实用的推理期自适应包装。

### 8.4 检查 contour 是否闭合

闭合不是只看：

- `z_current` 回到 `z_start` 附近

当前代码还会同时看：

- 已经走了足够多步
- 累积路径长度够长
- 距离起点曾经离开得够远
- 绕锚点的角度累积够多

满足这些条件才认为真闭合，而不是数值上的局部小回头。

## 9. 当前版本里哪些东西已经被删掉

从代码真实状态看，下面这些已经不在主链里了：

### 9.1 restart 分支

已经没有：

- restart 标签
- restart head
- restart loss
- restart 相关训练/推理控制逻辑

### 9.2 NN 主链里的多 ODE 方法切换

当前推理主链已经不是：

- rk4 / heun / tangent 多方法切换

而是：

- 单一 tangent tracker

### 9.3 推理主链里的缩步重试机制

当前推理主链不会再做：

- 多轮缩步重试
- 额外记录缩步次数字段
- 用缩步次数反馈控制器

所以如果你现在看当前代码，推理主链的真实结构就是：

```text
网络给步长
-> 切向 predictor
-> 近似接受 / 延迟纠偏 / 局部投影 / exact SVD / 全局投影
-> 接受该步
```

## 10. 为什么这个算法明明还会做 SVD，却仍然可能更快

这是你提的最关键问题之一。

你的问题可以重述成：

> 传统算法的 bottleneck 是 SVD 很慢；切向或梯度本身接近 O(1)。
> 现在我的方法也没有把 SVD 消掉，只是“传统数值 + 神经网络预测步长”，为什么还能有效？

答案是：**因为收益不来自把“单次 SVD”变便宜，而来自改变“整条 rollout 里昂贵步骤出现的频率和总步数”。**

具体分四层看。

### 10.1 切向本来就不是瓶颈

这一点你说得对。

当前切向公式：

\[
\frac{dz}{ds}= i\frac{v^*u}{|v^*u|}
\]

本身计算量极小。

所以“学方向”本来就不是重点，甚至可能没有意义。

这也是当前代码为什么已经收缩到：

- 白盒几何管方向
- 网络只管步长

### 10.2 真正影响总耗时的是“要走多少步”

如果一个策略总是很保守，那么它会出现：

- 每步都很短
- 虽然单步很稳，但整条 contour 需要更多步

哪怕每一步的公式都一样，**总步数一多，总时间就上去了**。

网络如果能学到：

- 平滑区域可以放心放大步长
- 风险区域再收回来

那么同样一条 contour 就能用更少 accepted steps 走完。

这已经会直接降低 wall-clock。

### 10.3 真正贵的不是“有 SVD”，而是“高频无差别地做 SVD”

当前 tracker 并没有说：

- 永远不做 SVD

而是说：

- 先用更便宜的信息判断能不能接受
- 必要时才升级到精确刷新

也就是说，优化点是：

- **减少不必要的 SVD 触发频率**
- **把 SVD 尽量留给真正危险的状态**

这和“把 SVD 算法本身改快”完全是两件事。

### 10.4 更好的步长控制会减少额外修正

如果步长选得不合适，会出现：

- projection 更频繁
- 局部漂移更大
- 轨迹抖动更多
- 闭合前要走冤枉路

而一个好的步长策略会让 rollout 更像：

- 该大就大
- 该稳就稳
- 不把状态经常送进必须重修正的区域

所以它省下来的不只是“步数”，还有：

- projection 次数
- 精确刷新次数
- 不必要的纠偏开销

### 10.5 所以真正的收益来源是什么

一句话说清楚：

> 你的方法的收益来自“更少的步数 + 更少的不必要昂贵修正”，而不是来自“把单次 SVD 算法改成更快复杂度”。

所以它有效并不矛盾。

---

## 11. 一句话总结

当前代码最准确的描述是：

> **用 8 维局部状态让网络预测步长，用白盒切向公式负责方向，用单一 tangent tracker 执行 contour 追踪；tracker 优先走便宜近似路径，只在必要时才升级到局部投影、精确 SVD 和全局投影。**

如果再压缩成更短一句：

> **你现在的算法学的不是“怎么求 contour”，而是“什么时候可以走快、什么时候必须保守”，而真正节省下来的，是整条 rollout 里的昂贵修正次数和总步数。**
